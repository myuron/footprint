# footprint 要件定義書

> 本書は footprint(Claude Code セッションログ閲覧 TUI)の要件を定義する。
> JSONL の構造仕様は [`jsonl-format.md`](./jsonl-format.md) を参照。

## 1. 概要・目的

- **背景**: Claude Code とのやり取りを後から確認するには `~/.claude/projects/` 配下の JSONL を読む必要があるが、この形式は機械可読で、人間には非常に読みづらい。
- **主目的**: セッションログ(JSONL)を、人間が見やすい構造化された形で表示し、ユーザのデバッグを助ける。**「JSONL のままでは読みづらい」を解消することが最優先**。
- **副目的**(MVP外・将来 → §6): 意図したタイミングで skills / hooks / subagent などが発火しているかを確認できるようにする。
- **想定読者**: 将来のコントリビュータ、および実装を担う Claude。本書は footprint の実装の指針となる仕様である。

## 2. スコープ

### 2.1 MVP に含む

- カレントディレクトリのプロジェクトに属するセッションの一覧表示(§4.1)
- セッション内容の構造化表示: 現在の会話の一本道復元、プロンプト単位ブロック、折りたたみ、ツールの 1 行表示(§4.2)
- 全画面 2 画面のレイアウト(§4.3)と、キーボードのみで完結する vim ライクな操作(§4.4)
- ローカルのタイムゾーン・ロケールに合わせた時刻表示(§5)
- 手動リロードによる再読み込み(§5)

### 2.2 MVP に含まない(将来対応 → §6)

- 複数プロジェクトの横断表示
- 会話の分岐(branch)の表示
- 圧縮前履歴の表示(圧縮境界より前の会話)
- 副目的(hooks / subagent / skill の発火可視化)
- サブエージェントログ(`subagents/`)の閲覧
- 非会話レコードの表示トグル
- リアルタイム追従(ファイル監視による自動更新)
- 検索・フィルタ
- Emacs バインド、およびキーバインドのユーザによる変更
- Windows 対応(まずは Linux / macOS)

### 2.3 恒久的にスコープ外

- JSONL の編集・書き換え。footprint は**常に閲覧専用**であり、セッションログを変更しない。

## 3. 前提・用語

JSONL の詳細な構造は [`jsonl-format.md`](./jsonl-format.md) を参照。本書で用いる主な用語:

- **セッション**: 1 つの `<session-uuid>.jsonl` ファイル。1 行 = 1 レコード(イベント)で、追記専用。
- **会話ツリー**: レコードが `parentUuid` → `uuid` の親子で構成する木構造。中断・再送で分岐しうる。
- **現在の会話**: `last-prompt` の `leafUuid` から `parentUuid` を遡って復元される一本道の会話(§4.2.1)。
- **プロンプト単位ブロック**: 「人が入力した 1 つのユーザプロンプト」と、それに紐づく一連のやりとりをまとめた表示単位(§4.2.2)。
- **圧縮境界(compact_boundary)**: 会話圧縮により `parentUuid` チェーンが途切れる地点(§4.2.1)。
- **非会話レコード**: 人の会話・ツールのやりとり以外のレコード(`attachment`、`system`、`mode` / `permission-mode` 等の状態系、メタ系、`isMeta` な `user` 等)。MVP では非表示(§4.2)。
- **寛容パース**: 未知のフィールド・未知の `type` を許容し、壊れた行があっても可能な範囲を表示する解析方針(§5)。

## 4. 機能要件

footprint は 2 つの画面(セッション一覧 / セッション内容)を持つ。両画面ともキーボードのみで操作が完結する(§4.4)。

### 4.1 セッション一覧画面

起動時の初期画面。**カレントディレクトリのプロジェクト**に属するセッションを一覧表示する。

- **対象の特定**: 起動時の作業ディレクトリ(cwd)の絶対パスを `/`・`.` → `-` でエンコードし、`~/.claude/projects/<エンコード済みパス>/` 配下の `*.jsonl` を対象とする。
- **表示する列**:
  - 最終更新時刻(ファイルの最終更新。ローカル時刻で表示)
  - セッションID(ファイル名の UUID)
  - タイトル(「何をしたセッションか」が分かるもの。`custom-title` > `ai-title` > `slug` の優先で採用。いずれも無ければタイトルは空とし、セッションID で識別する)
    - 注: `custom-title` / `ai-title` は独立したレコードとして追記され、**最後に現れた値が現在値**となる。取得のためのファイル走査コストと §5 の性能要件を両立させる方法は調査事項とする(→ §7)。
- **並び順**: 最終更新が新しい順(降順)。
- **選択**: 任意のセッションを選択(Enter)すると、そのセッションのセッション内容画面(§4.2)へ遷移する。
- サブディレクトリ(`<session-uuid>/subagents/` 等)は一覧に含めない。
- **空状態**: 対象ディレクトリが存在しない、または対象の `*.jsonl` が 0 件の場合は、その旨を示すメッセージを表示する(異常終了しない)。

### 4.2 セッション内容画面

選択したセッションの会話を、人間が読みやすい構造で表示する画面。本ツールの中心機能。

- **表示対象**: 人の会話(`user` / `assistant`)とツールのやりとりを表示する。**非会話レコード**(`attachment`、`system`、`isMeta` で内部注入された `user`、`mode` / `permission-mode` などの状態系レコード)は **MVP ではデフォルト非表示**とする(表示トグルは将来 → §6)。
- **空状態**: 復元した会話が 0 件(表示すべき会話レコードが無い)の場合は、その旨を示すメッセージを表示する(異常終了しない)。

#### 4.2.1 表示する会話の範囲

- JSONL は中断・再送により会話が分岐しうるが、MVP では **「現在の会話」を一本道で表示**する(分岐は表示しない → §6)。
- 「現在の会話」は、**最後に現れた `last-prompt` レコード**の `leafUuid` を葉として、`parentUuid` を親方向に遡って復元する(`last-prompt` はファイル内に複数回追記されるため、最後の値を現在値とする)。
- 会話の圧縮(compaction)により、`compact_boundary` レコードで `parentUuid` チェーンが途切れる。MVP では **圧縮境界で打ち切り、境界以降(圧縮後)の会話のみを表示**する(圧縮前の履歴を `logicalParentUuid` で辿る対応は将来 → §6)。圧縮要約(`isCompactSummary` な `user`)、および圧縮境界以降で最初の人間プロンプト(§4.2.2)より前に位置するレコードは、MVP では表示しない。
- **フォールバック(MVP 暫定)**: `last-prompt` が存在しない、または `leafUuid` が現ファイルに存在しない(resume 時に元セッションの uuid を指す)場合は、**ファイル内の最後の会話レコードを葉とみなして**遡る。

#### 4.2.2 プロンプト単位ブロック

- 会話は **ユーザプロンプトを起点とするブロック**の連なりとして表示する。1 ブロック = 「1 つのユーザプロンプト」+「それに紐づく一連のやりとり(Claude の応答・ツール呼び出し・結果など)」。
- **ブロックの起点**は「人が実際に入力したユーザプロンプト」= `user` レコードのうち **人間入力を表す `promptSource`**(`typed`、および Claude 動作中にキュー投入された `queued` 等)を起点とする。ツール実行結果を運ぶ `user`、`isMeta`(内部注入)、`isCompactSummary`(圧縮要約)、`promptSource: "sdk"` などは起点にしない。`queued` を起点から漏らすと、実在する人間プロンプトが前のブロックに吸収されてしまう点に注意。
  - 補足: `origin.kind: "human"` も人間入力の判定に使える。どの値を主判定にするか(および `suggestion_accepted` の扱い)は実装時に実データで確認する(→ §7.1)。
  - スラッシュコマンド(`<command-name>` 形式で入力されたもの)は、コマンド名をブロックの要約に用いる。
- **ブロックの範囲**は、ある起点ユーザプロンプトから次の起点ユーザプロンプトの直前までを、復元済みの `parentUuid` チェーン上で束ねる。
  - 注: `promptId` は `user` レコードにしか付かず、`assistant` 等には付かないため、ブロックの束ねキーには使えない。復元済みチェーンの並び順で区切る。
- **初期表示ではブロック単位で折りたたむ**。各ブロックはユーザプロンプトの要約(先頭行など)と時刻を 1 行で見せる。
- ブロックを展開すると、内部のやりとりが時系列で表示される。

#### 4.2.3 ブロック内の行と折りたたみ

展開後、ブロック内の各要素は 1 行のサマリで表示し、必要に応じてさらに展開できる。

- **行の単位**: 表示上の 1 行は **content ブロック単位**(`text` / `thinking` / `tool_use` それぞれを 1 行)とする。1 応答が同一 `message.id` で複数レコードに分割される場合があるため、**同一 `message.id` は 1 応答としてまとめて**扱う(重複行にしない)。
- **行の基本形**: `[ローカル時刻] [役割/種別] [1 行サマリ] [右端に補助情報]`。右端の補助情報は成否など任意の要素とし、MVP での必須項目とはしない(具体は実装時に定める)。
- **ツール呼び出しは 1 行で中身が分かること**(必須要件):
  - `Read` → 読んだファイルパス
  - `Bash` → 実行コマンド
  - `Skill` → 呼び出したスキル名
  - その他のツール → ツール名 + 引数の要点
- **ツール結果**: `tool_use.id` ⇔ `tool_result.tool_use_id` で対応付け、対応するツール呼び出し行の直下に従属表示する。要約 + 行数などで折りたたみ、展開で全文。成否の表示は結果レコードの `is_error` 等から判定する(Bash の結果に `exitCode` は含まれないため、終了コードそのものの表示はしない。判定手段の詳細は → §7)。
- **thinking(思考)**: 行数を示して折りたたみ表示し、展開で全文。
- **長いテキスト**: 先頭数行のみ表示し、展開で全文。
- 展開はその場でのインライン展開(アコーディオン)とする。

### 4.3 画面レイアウト

- 両画面とも**全画面**を使い、画面遷移は「セッション一覧 → セッション内容」および戻る操作で構成する(左右分割ペインではない)。
- 各画面は以下で構成する:
  - **ヘッダ**: 現在の画面名・対象(一覧ではプロジェクトパス、内容ではセッションのタイトル/ID)。
  - **本文**: 一覧は行のリスト、内容はブロック/行のリスト。縦スクロールし、選択位置(カーソル)を反転等で明示する。
  - **フッタ**: 主要キーのヒントを表示する。
- 一覧の列幅、本文の折り返し/省略の詳細は実装時に定める(→ §7)。

### 4.4 操作・キーバインド

- **キーボードのみで全操作が完結する(必須要件)**。マウス操作に依存しない。
- MVP では **vim ライクなキーバインド**を提供する。MVP の確定割当は以下(将来はユーザ変更・Emacs バインドに対応 → §6):

| キー        | セッション一覧       | セッション内容                |
| ----------- | -------------------- | ----------------------------- |
| `j` / `k`   | カーソル下 / 上      | カーソル下 / 上               |
| `g` / `G`   | 先頭 / 末尾          | 先頭 / 末尾                   |
| `Enter`     | 選択セッションを開く | カーソル行の展開 / 折りたたみ |
| `Esc` / `q` | アプリ終了           | 一覧に戻る(一覧では終了)      |
| `r`         | 一覧を再読み込み     | セッションを再読み込み        |

- 上表はキーバインドの**規範(確定仕様)**であり、単なる例示ではない。上表にないキー(ページスクロール等)は実装時に追加してよい。

## 5. 非機能要件

- **性能**: 一覧画面・内容画面ともに、操作に対して画面が返るまで **3 秒以内**。1 セッション 1.6 万レコード級のファイルでもこの要件を満たす(必要なら遅延読み込み等で対応)。
- **寛容パース**: JSONL のフォーマットはバージョンごとに変化しうるため、パーサは **未知のフィールド・未知の `type` を許容**する(`#[serde(deny_unknown_fields)]` は使わない)。パース不能な行があっても異常終了せず、解析できた範囲を表示する。
- **時刻・ロケール**: タイムスタンプは閲覧環境のローカルタイムゾーン・ロケールに合わせて表示する(海外ユーザの利用を想定)。
- **対応 OS**: Linux / macOS を先行対応する。Windows は将来対応(→ §6)。
- **閲覧専用**: JSONL ファイルを書き換えない(§2.3)。
- **更新方針**: 起動時に対象を読み込み、**手動リロードキー**で再読み込みする。閲覧中に追記された内容はリロードで反映する。ファイル監視による自動追従はしない(→ §6)。

## 6. 将来拡張

MVP では実装しないが、将来的に取り込みたい機能:

- **副目的の実現**: skills / hooks / subagent の発火の可視化(意図したタイミングで発火したかの確認)。
  - Hook 発火は `attachment`(`hook_success` 等)や `system`(`stop_hook_summary`)に記録される。ただし出力を出さない無言フックの検出可否は要調査(→ §7)。
- **会話の分岐(branch)の表示**: 中断・再送で生じた枝を辿れるようにする。
- **圧縮前履歴の表示**: `logicalParentUuid` を辿って圧縮境界より前の会話も表示する。
- **リアルタイム追従**: ファイル監視による自動更新。
- **検索・フィルタ**: セッション/会話内の検索、種別によるフィルタ。
- **キーバインド**: Emacs バインドの提供、およびユーザによる任意のキーバインド変更。
- **対応 OS の拡大**: Windows 対応。
- **プロジェクト横断**: カレント以外のプロジェクトのセッションも一覧・閲覧。
- **サブエージェントログの閲覧**: `<session-uuid>/subagents/` 配下の会話の表示。
- **非会話レコードの表示トグル**: MVP で非表示にしたレコードを任意で表示する切り替え。

## 7. 未解決・調査事項

### 7.1 MVP 実装中に詰める(暫定仕様はあるが実データで要確認)

- **ブロック起点の判定**: `promptSource: "typed"` を主判定とするか `origin.kind: "human"` とするか。スラッシュコマンドやその他の `promptSource` 値の扱い(§4.2.2)。
- **ツール結果の成否判定**: Bash 等の結果に終了コードが含まれないため、`is_error` 等で成否を判定する方法の妥当性・信頼性(§4.2.3)。
- **タイトル・時刻の取得コスト**: `custom-title` / `ai-title` / `slug` は走査が必要。全行走査を避ける方法(末尾からの逆読み等)と §5 の 3 秒要件の両立(§4.1)。
- **表示の具体値**: ブロックサマリや「先頭数行のみ表示」の行数、1 行サマリが端末幅を超える場合の省略規則、一覧の列幅配分(§4.1〜§4.3)。
- **エッジな会話レコードの表示**: API エラー応答(`isApiErrorMessage`)、ユーザ割り込み、`user` の `image` ブロックなどの見せ方(MVP はベストエフォート)。

### 7.2 将来機能(§6)向けの調査

- **無言フックの検出可否**: 出力(systemMessage 等)を出さないフックの発火を JSONL から漏れなく検出できるか。
- **許可プロンプトの可視化**: あるツール呼び出しが Ask(許可プロンプト)を伴ったかを JSONL から判定できるか。`settings.json` 修正の手がかりに使えるとよいが、実現性は未確認。
